# Fully-Dressed Use Case Template and Writing Guide

## Purpose

This reference provides the standard template for fully-dressed use case descriptions, writing guidelines, and worked examples. All use cases generated by the Use Case Modeling skill shall conform to this template.

## Reference Standards

- UML 2.5.1 -- Use case notation and relationships
- IEEE 29148-2018 -- Behavioral requirements specification
- Dennis, Wixom, Tegarden -- "Systems Analysis and Design with UML" (fully-dressed format)

## Use Case Levels

### User-Goal Level (Primary)

The system shall model use cases at the user-goal level. A user-goal use case represents a complete interaction that delivers measurable value to the primary actor. Examples: "Process Payment", "Register Account", "Submit Order".

### Subfunction Level (Supporting)

Subfunction use cases represent shared behaviors included by multiple user-goal use cases. They exist only to reduce duplication via `<<include>>` relationships. Examples: "Authenticate User", "Validate Address", "Log Audit Event".

**Rule:** Do not create standalone subfunction use cases. Every subfunction shall be included by at least one user-goal use case.

## Complete Template

```markdown
### UC-[NNN]: [Verb-Noun Name]

| Field | Value |
|-------|-------|
| **UC-ID** | UC-[NNN] |
| **Name** | [Verb-Noun Name] |
| **Scope** | [System Name] |
| **Level** | User Goal / Subfunction |
| **Primary Actor** | [Actor who initiates the use case] |
| **Stakeholders & Interests** | [Stakeholder]: [What they want from this use case] |
| **Preconditions** | [Verifiable conditions that must be true before execution begins] |
| **Success Guarantee (Postcondition)** | [System state guaranteed after successful completion] |
| **Minimal Guarantee** | [System state guaranteed even if the use case fails] |
| **Priority** | Critical / High / Medium / Low |
| **Frequency of Use** | [Quantified: e.g., 500/day, 20/week, On-demand] |
| **Trigger** | [Specific event or actor action that starts this use case] |

#### Main Success Scenario (Basic Flow)

1. The [Primary Actor] [initiates action with specific data].
2. The system shall [validate/process the input].
3. The system shall [perform business logic].
4. The [Primary Actor] [reviews/confirms result].
5. The system shall [persist state change].
6. The system shall [confirm completion to the actor].

#### Alternative Flows (Extensions)

**[Step]a. [Guard Condition]:**
1. The system shall [alternative processing].
2. Resume at step [N].

**[Step]b. [Different Guard Condition]:**
1. The system shall [different alternative].
2. The use case ends.

#### Exception Flows (Error Handling)

**[Step]a. [Error Condition]:**
1. The system shall [log the error with context: actor, timestamp, input data].
2. The system shall [display actionable error message to the actor].
3. The system shall [rollback/compensate any partial state changes].
4. Resume at step [N] / The use case aborts.

#### Business Rules

- BR-[NNN]: [Rule name] -- [Rule description from business_rules.md]

#### Data Requirements

| Entity | Operation | Attributes |
|--------|-----------|------------|
| [Entity Name] | Create / Read / Update / Delete | [Key attributes involved] |

#### Open Issues

- OI-[NNN]: [Unresolved question requiring stakeholder input]
```

## Writing Guidelines

### Active Voice Rule

Every step shall use active voice:
- Correct: "The system shall calculate the total."
- Incorrect: "The total is calculated by the system."

### Stimulus-Response Pairing

Alternate between actor actions (stimulus) and system responses:
- Step 1: Actor does something.
- Step 2: System responds.
- Step 3: Actor does the next thing.
- Step 4: System responds.

**Rule:** Two consecutive actor steps or two consecutive system steps indicate a missing interaction. Split or merge as needed.

### Technology Neutrality

Do not embed UI details in the main success scenario:
- Correct: "The Customer provides payment credentials."
- Incorrect: "The Customer clicks the 'Pay Now' button and enters credit card number in the text field."

Reference the Interface Specification (Section 3.1) for UI-specific details.

### Precondition Rigor

Every precondition shall be verifiable and traceable:
- Correct: "The Customer has an active account with status = 'Active' in the Customer table."
- Incorrect: "The Customer is logged in."

### Postcondition Completeness

State both success guarantee and minimal guarantee:
- **Success Guarantee**: "The order is persisted with status = 'Confirmed' and the inventory count is decremented by the ordered quantity."
- **Minimal Guarantee**: "No payment is charged. The order remains in status = 'Draft'."

## Example 1: Simple CRUD Operation

### UC-010: Create Customer Account

| Field | Value |
|-------|-------|
| **UC-ID** | UC-010 |
| **Name** | Create Customer Account |
| **Scope** | Registration Subsystem |
| **Level** | User Goal |
| **Primary Actor** | Prospective Customer |
| **Stakeholders & Interests** | Marketing: wants demographic data for segmentation. Compliance: wants consent records for GDPR. |
| **Preconditions** | The Prospective Customer does not have an existing account (email is unique in Customer table). |
| **Success Guarantee** | A new Customer record exists with status = 'Active'. A welcome notification is queued. |
| **Minimal Guarantee** | No partial Customer record exists. No notification is sent. |
| **Priority** | Critical |
| **Frequency of Use** | 200/day |
| **Trigger** | The Prospective Customer requests account creation. |

#### Main Success Scenario

1. The Prospective Customer provides name, email, and password.
2. The system shall validate that the email format conforms to RFC 5322.
3. The system shall verify that no existing Customer record uses the provided email.
4. The system shall hash the password using bcrypt with a cost factor of 12.
5. The system shall persist the Customer record with status = 'Active' and creation timestamp.
6. The system shall queue a welcome notification to the provided email.
7. The system shall confirm account creation to the Prospective Customer.

#### Alternative Flows

**3a. Email already exists:**
1. The system shall inform the Prospective Customer that the email is already registered.
2. The system shall offer account recovery. Resume at step 1.

#### Exception Flows

**5a. Database write failure:**
1. The system shall log the error with actor context and timestamp.
2. The system shall display a retry message to the Prospective Customer.
3. The use case aborts. Minimal guarantee holds.

## Example 2: Complex Multi-Step Workflow

### UC-025: Process Return and Refund

| Field | Value |
|-------|-------|
| **UC-ID** | UC-025 |
| **Name** | Process Return and Refund |
| **Scope** | Order Management System |
| **Level** | User Goal |
| **Primary Actor** | Customer |
| **Stakeholders & Interests** | Customer: wants money back promptly. Finance: wants accurate ledger entries. Warehouse: wants returned item inspected. |
| **Preconditions** | The Customer has an order with status = 'Delivered' and the return window (30 days) has not expired. |
| **Success Guarantee** | The order item status = 'Returned'. Inventory is incremented. Refund is issued to the original payment method. Ledger entry is created. |
| **Minimal Guarantee** | No refund is issued. The order item status remains unchanged. |
| **Priority** | High |
| **Frequency of Use** | 50/day |
| **Trigger** | The Customer initiates a return request for a delivered order item. |

#### Main Success Scenario

1. The Customer selects the order item to return and provides a return reason.
2. The system shall validate that the order item is eligible for return (status = 'Delivered', within 30-day window).
3. The system shall generate a Return Authorization (RA) number and update the order item status to 'Return Initiated'.
4. The system shall provide the RA number and shipping instructions to the Customer.
5. The Warehouse Staff receives the returned item and records the inspection result.
6. The system shall update the order item status to 'Return Received'.
7. The system shall increment inventory count for the returned SKU.
8. The system shall initiate refund to the original payment method.
9. The system shall create a ledger entry for the refund amount.
10. The system shall notify the Customer that the refund has been processed.

#### Alternative Flows

**2a. Return window expired:**
1. The system shall inform the Customer that the return window has closed.
2. The use case ends.

**5a. Item fails inspection (damaged by customer):**
1. The Warehouse Staff records the item as 'Damaged - Customer Fault'.
2. The system shall notify the Customer that a partial refund (50%) will be issued.
3. Resume at step 8 with adjusted refund amount.

#### Exception Flows

**8a. Payment gateway unavailable:**
1. The system shall log the failure with RA number and payment details.
2. The system shall queue the refund for retry in 1 hour.
3. The system shall notify the Customer of the delay.
4. The use case pauses; resume at step 8 upon retry.

#### Business Rules

- BR-012: Return Window -- Items shall be eligible for return within 30 calendar days of delivery date.
- BR-013: Damaged Return Policy -- Items damaged by the customer shall receive a 50% refund.
- BR-014: Refund Method -- Refunds shall be issued to the original payment method only.

#### Data Requirements

| Entity | Operation | Attributes |
|--------|-----------|------------|
| Order Item | Update | status, return_reason, ra_number |
| Inventory | Update | quantity (increment) |
| Ledger | Create | amount, type='refund', reference=RA number |
| Payment | Create | refund_amount, method, status |
