# Test Environments

## 4.1 Environment Summary

| Environment | Purpose | Database | Data | Reset Cycle |
|---|---|---|---|---|
| Development (local) | Unit tests and integration tests | MySQL test schema, isolated per developer workstation | Seeded fixtures generated by seed scripts | Per test class (transactional rollback or schema rebuild) |
| Staging | System tests, UAT, performance tests, security tests | MySQL 9.1 production-equivalent schema and engine version | Anonymised production clone or volume-equivalent synthetic data | Full reset weekly; partial reset on demand |
| Production | Smoke tests only, executed immediately post-deployment | Production (live) | Live tenant data | Never reset — read-only smoke probes only |

## 4.2 Development Environment

### 4.2.1 Configuration

- Every developer runs a local Apache + PHP 8.3 + MySQL 9.1 stack matching the production versions.
- The test database schema is a separate named database (e.g., `longhorn_erp_test`) distinct from the development database (`longhorn_erp_dev`).
- PHPUnit is configured via `phpunit.xml` to use the test database. The `APP_ENV=testing` environment variable must be set before any test run.

### 4.2.2 Data Seeding

- Seed scripts reside in `database/seeders/testing/` and are invoked by `php artisan db:seed --env=testing` (or the equivalent framework command).
- Seed scripts must be deterministic: the same seed script run twice on an empty schema produces identical data.
- Each integration test class wraps its database interactions in a transaction that is rolled back in `tearDown()`, ensuring test isolation without requiring a full schema rebuild between tests.

### 4.2.3 Secrets Management

- Database credentials, API keys, and JWT secrets are never committed to the repository.
- All secrets for the test environment are loaded from `.env.testing`, which is listed in `.gitignore`.
- CI/CD pipelines inject secrets via environment variables from the secrets manager.

## 4.3 Staging Environment

### 4.3.1 Configuration

- Staging mirrors the production server specification: Apache, PHP 8.3, MySQL 9.1, identical `php.ini` settings.
- SSL/TLS is enabled on staging using a valid certificate so that JWT and CSRF tests execute under realistic transport security conditions.
- The staging environment is accessible only via VPN or allowlisted IP ranges; it is not publicly routable.

### 4.3.2 Tenant Isolation Requirement

- Staging always contains a minimum of 2 test tenants (`tenant_id = 1` and `tenant_id = 2`) with distinct, non-overlapping data sets.
- Before every system or security test cycle, a verification script asserts that no record in any module table for `tenant_id = 1` shares a primary key with a record for `tenant_id = 2`.
- Any cross-tenant access attempt (authenticated as Tenant A, requesting a resource belonging to Tenant B) must return HTTP 403 or an empty result set. A response containing Tenant B's data is a Critical defect that halts the test cycle immediately.

### 4.3.3 Performance Test Data Volume

- The staging database must be populated to the volume thresholds specified in Section 3.3.2 before performance tests execute.
- The anonymised production clone is produced by the `scripts/anonymise-dump.sh` script, which replaces all personally identifiable fields (names, phone numbers, national IDs, bank account numbers) with synthetic equivalents before import into staging.

### 4.3.4 Reset Schedule

- A full staging environment reset (schema drop and re-import from the latest anonymised clone) is scheduled every Sunday at 02:00 EAT.
- An on-demand reset can be triggered by the QA lead via the `scripts/reset-staging.sh` script. The reset log is archived in the project's operations log.

## 4.4 Production Environment

### 4.4.1 Post-Deployment Smoke Tests

- Immediately after every production deployment, the QA lead executes a documented smoke test script that covers:
  1. Login succeeds for a designated smoke-test account.
  2. Dashboard loads within 3 s (manual measurement).
  3. A read-only GL report query returns a non-empty result set.
  4. Mobile API health endpoint returns HTTP 200 with the correct build version string.
- If any smoke test assertion fails, the deployment is rolled back immediately.

### 4.4.2 Data Safety Constraints

- No test data is written to the production database under any circumstances.
- The smoke-test account is a read-only system user with no write permissions. It cannot create, modify, or delete any record.
- Production database credentials are never loaded in any developer's local environment.
