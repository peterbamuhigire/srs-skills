# 4. Non-Functional Requirements — Phase 5 People

## 4.1 Performance

**NFR-P5-001**
The payroll computation for a batch of up to 200 employees shall complete within 30 seconds from initiation to summary display on the web ERP. Measurement: timed end-to-end in a PHPUnit integration test with a 200-employee seed dataset.

**NFR-P5-002**
The HR Self-Service App home screen shall load within 3 seconds on a 4G LTE connection (minimum 10 Mbps downlink). Measurement: measured from JWT authentication response to screen render using Android Systrace.

**NFR-P5-003**
Biometric attendance import for a batch of 500 records shall complete within 30 seconds. Measurement: import timed with a test file containing 500 synthetic attendance records.

## 4.2 Reliability

**NFR-P5-004**
The payroll GL auto-posting shall succeed or fully roll back in a single database transaction. Partial GL posting (some entries committed, some not) is architecturally impossible. Measurement: MySQL InnoDB transaction rollback test.

**NFR-P5-005**
Leave balance computations shall be accurate to the nearest half-day. A PHPUnit test suite shall cover leave balance edge cases: year boundary, carry-forward maximum, concurrent approved applications.

## 4.3 Security

**NFR-P5-006**
All HR and payroll REST API endpoints shall validate the JWT Bearer token and enforce RBAC at the API layer. Penetration test in Phase 7 (F-019) shall include attempts to access payroll endpoints without valid tokens and with valid tokens of insufficient role, confirming HTTP 403 responses.

**NFR-P5-007**
Employee NIN fields shall be encrypted at rest. The application database user shall not hold the encryption key directly; keys shall be managed via the application `.env` configuration (DC-006).

## 4.4 Maintainability

**NFR-P5-008**
All payroll computation logic (PAYE, NSSF, LST, overtime) shall achieve a minimum 80% PHPUnit code coverage. Coverage report shall be generated on every GitHub Actions CI run.

**NFR-P5-009**
All payroll element processing shall be implemented via the Service/Repository pattern (PHP-DI). Adding a new statutory deduction type (e.g., a new tax introduced by Parliament) shall require only a new configuration record and a new service class; no modification to existing payroll run logic shall be required (DC-002, DC-007).

## 4.5 Usability

**NFR-P5-010**
A newly hired HR assistant shall be able to process a complete leave application approval workflow (receive notification, review application, approve, confirm deduction) without reading a manual, satisfying DC-001. Acceptance criterion: usability walkthrough with one HR assistant who has not been trained on the system, completed in under 5 minutes with 0 support interventions.

## 4.6 Data Integrity

**NFR-P5-011**
The payroll database schema shall enforce the following integrity constraints: employee ID foreign key on every payroll record; non-null pay period on every payroll run; unique constraint on (employee_id, pay_period, payroll_run_id) on payroll detail records; check constraint that net pay is not negative before a payroll run is marked "Approved".

# 5. External Interface Requirements

## 5.1 ZKTeco Biometric Device Interface

The system shall connect to ZKTeco fingerprint attendance devices via the device's SDK or export file interface. The exact integration method (SDK, TCP/IP direct connection, or file import) shall be confirmed once device model numbers are provided by BIRDC IT. `[CONTEXT-GAP: GAP-005]` Regardless of integration method, attendance data must transfer into the HR module without manual re-entry (BR-016).

## 5.2 Mobile Money Interface (Casual Worker Salary)

The system shall integrate with the MTN MoMo Business API and Airtel Money API for bulk casual worker salary disbursement. API credentials shall be stored in the system `.env` configuration, not in the database, and managed via the IT Administration panel (F-017). `[CONTEXT-GAP: GAP-002]`

## 5.3 PHPMailer / WhatsApp Interface

Payslip PDF files generated by mPDF shall be delivered via PHPMailer (SMTP TLS) for email delivery or via a configured WhatsApp Business API connector for WhatsApp delivery. The employee's delivery preference shall be configurable per employee record.

## 5.4 Android HR Self-Service App Interface

The HR Self-Service App communicates with the web ERP via a dedicated REST API using JWT Bearer token authentication. All HR self-service endpoints shall enforce the same RBAC rules as the web interface. The app uses Android WorkManager for background leave application sync when offline.

## 5.5 General Ledger Interface

Payroll GL auto-posting calls the internal GL posting service (F-005) via a PHP service interface. The HR/Payroll module has no direct database writes to GL tables; all GL entries are mediated through the GL service class, maintaining architectural separation and ensuring hash chain integrity (BR-013).
