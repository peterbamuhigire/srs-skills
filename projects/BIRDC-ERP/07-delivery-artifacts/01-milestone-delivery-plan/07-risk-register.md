## 6. Risk Register Summary

The following table documents the top 5 risks to milestone delivery. Each risk corresponds to a known gap or constraint identified during the requirements phase. All risks must be reviewed at each milestone kickoff.

| Risk ID | Description | Probability | Impact | Linked Gap | Mitigation |
|---|---|---|---|---|---|
| RSK-001 | URA EFRIS API sandbox credentials not obtained before Phase 1 invoice testing | High | High — blocks M-001 sign-off | GAP-001 | BIRDC IT to apply for EFRIS sandbox credentials immediately. Peter Bamuhigire to follow up with URA liaison. EFRIS integration can be mocked in test environment until credentials are received, but live sign-off requires real API. |
| RSK-002 | Legacy accounting data migration is more complex than anticipated, causing go-live delay | Medium | High — delays M-007 and production cut-over | GAP-014 | BIRDC Finance to provide a complete inventory of existing accounting software and data by M-002 kickoff. Data migration scope and effort to be assessed no later than M-003 and, if significant, a Change Request raised. |
| RSK-003 | Intermittent internet connectivity at Nyaruzinga delays integration testing and developer collaboration | High | Medium — increases development time and may delay demonstrations | None (known infrastructure constraint) | Development environment configured for offline operation. All integration tests that require external APIs (EFRIS, MTN MoMo) are scheduled for periods of confirmed connectivity. Developer team uses offline-capable tooling. |
| RSK-004 | Qualified PHP/Laravel developers available in Uganda are insufficient to staff the project at required velocity | Medium | High — directly affects milestone delivery dates | None (labour market constraint) | Peter Bamuhigire to begin developer recruitment no later than 4 weeks before M-001 kickoff. Remote developers considered if local talent pool is insufficient. Developer capacity is reviewed at each milestone kickoff and the delivery schedule adjusted if necessary, with Director notification. |
| RSK-005 | PPDA compliance documentation requirements are not fully confirmed before procurement module is built | Medium | Medium — rework required if thresholds or document formats change | GAP-007 | BIRDC Administration Officer to obtain the current PPDA threshold schedule from PPDA Uganda and confirm applicable categories for BIRDC/PIBID before M-003 kickoff. Peter Bamuhigire to design the procurement approval matrix as configuration-driven (DC-002) so thresholds can be updated without code changes. |
