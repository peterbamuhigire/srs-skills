# Section 4: Non-Functional Requirements for the Restaurant/Bar Add-on Module

Non-functional requirements (NFRs) in this section apply exclusively to the F-011 module. General platform NFRs (availability, security, multi-tenancy isolation) defined for Phase 1 remain in effect and are not repeated here.

All performance thresholds in this section are measured under normal load, defined as: up to 10 concurrent KDS screens per tenant, up to 50 concurrent waiter sessions per branch, and up to 200 open orders per branch per service period.

---

## 4.1 Performance

**NFR-RES-001:** When a KOT is dispatched by a waiter, the system shall display the KOT on all KDS screens subscribed to the relevant station within 500 ms of the server confirming KOT creation. This latency is measured from server-side KOT creation timestamp to KDS client rendering, excluding client network propagation delays beyond the server boundary.

**NFR-RES-002:** When a table status changes (occupied, cleaning, available, reserved), the updated status shall propagate to all connected web and iOS clients displaying the table status board within 2 seconds of the change event being committed to the database.

**NFR-RES-003:** When a KOT is sent and triggers BOM stock deduction, the system shall complete all raw material inventory decrement operations and create the corresponding immutable stock movement records within 1 second of KOT send confirmation. KOT send confirmation shall not be presented to the waiter until the BOM deduction transaction is committed or until a shortfall warning is issued per FR-RES-065.

**NFR-RES-004:** When a bill is generated for an order with up to 50 KOT line items across multiple KOTs, the system shall render the complete itemised bill within 2 seconds.

**NFR-RES-005:** When a report is requested for a date range of up to 12 months, the system shall deliver the report data within 30 seconds for branches with up to 10,000 KOTs in the period.

---

## 4.2 Reliability

**NFR-RES-006:** When the KDS auto-refresh fails due to a transient network error, the system shall display a visible connectivity warning on the KDS screen and retry the connection automatically every 10 seconds. The KDS shall not display stale data without a visible "last updated" timestamp.

**NFR-RES-007:** When a waiter's iOS session loses connectivity mid-order, the system shall preserve all staged (unsent) KOT items in local device storage. On connectivity restoration, the waiter shall be able to dispatch the staged KOT. Offline KOT send is not supported; the system shall display an error if the waiter attempts to send a KOT without connectivity.

---

## 4.3 Usability

**NFR-RES-008:** When a waiter adds items to an order on iOS, the system shall require no more than 3 taps to add a menu item with no modifiers to the active KOT (tap category → tap item → tap "Add").

**NFR-RES-009:** When a waiter views the table status board on iOS, all tables shall be visible without horizontal scrolling for a standard iPhone display (375 pt width and above), using a scrollable grid layout.

---

## 4.4 Security

**NFR-RES-010:** When a KDS session is active on a shared kitchen display device, the system shall restrict the KDS browser session to read-only KOT display and item/KOT completion actions. The KDS session shall not expose order management, billing, inventory, or configuration functions.

**NFR-RES-011:** When a manager-level override is performed (e.g., table status override per FR-RES-009, tab credit limit override per FR-RES-043, KOT item void per FR-RES-026), the system shall require the manager to re-enter their PIN or password before the override is applied. The override event shall be recorded per BR-003.

---
