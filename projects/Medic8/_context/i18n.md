# Internationalisation Architecture — Medic8

## Supported Languages

| Code | Language | Status | Notes |
|------|----------|--------|-------|
| `en` | English | Primary — all strings mandatory | All modules; fallback target for all other locales |
| `fr` | French | Launch language — all strings required at ship | Targets DRC, Rwanda, and Francophone West Africa expansion |
| `sw` | Kiswahili | Launch language — all strings required at ship | Targets East Africa: Kenya, Tanzania, Uganda Swahili speakers |

No additional languages are added without a formal localisation sign-off that includes clinical terminology review by a native-speaker clinician.

## String Key Convention

All localisation keys follow the pattern `module.context.label`. Keys are lowercase, underscore-delimited, and never contain spaces or special characters.

Examples:

| Key | English value |
|-----|---------------|
| `opd.triage.blood_pressure_label` | Blood Pressure |
| `opd.triage.chief_complaint_label` | Chief Complaint |
| `pharmacy.dispensing.confirm_button` | Confirm Dispensing |
| `billing.invoice.total_amount_label` | Total Amount |
| `ai.documentation.approve_draft_button` | Approve Draft |
| `maternity.partograph.birth_attendant_label` | Birth Attendant |

Keys are never hardcoded in any UI layer. All strings are extracted through the localisation helper at render time.

## Locale Fallback Chain

The system resolves string keys in the following order:

1. The user's selected locale (`sw` or `fr`).
2. If the key does not exist in the selected locale: fall back to `en`.
3. If the key does not exist in `en`: render the key name itself (e.g., `opd.triage.blood_pressure_label`) and emit `[I18N-GAP: opd.triage.blood_pressure_label]` to the build log.

Machine translation is never used as a fallback. A missing string always falls through to English and is flagged for human translation. The fallback sequence is: `sw → en`, `fr → en`.

## Platform Implementation

### PHP / Laravel (Backend and Web Portal)

- Language files reside in `lang/en/`, `lang/fr/`, `lang/sw/` under the Laravel application root.
- Files within each locale directory are named by module: `lang/en/opd.php`, `lang/fr/opd.php`, `lang/sw/opd.php`.
- All UI strings are rendered using the Laravel `__()` helper: `__('opd.triage.chief_complaint_label')`.
- No string is ever hardcoded in a Blade template, controller response, or JSON API error message.
- The active locale is resolved from the authenticated user's `preferred_locale` profile field and set via `App::setLocale()` at the start of every request.

### Android

- String resources reside in `res/values/strings.xml` (English, default), `res/values-fr/strings.xml` (French), and `res/values-sw/strings.xml` (Kiswahili).
- All UI strings are referenced via `getString(R.string.<key>)` or the `stringResource()` Compose function.
- Android's resource resolution system provides automatic fallback to the default `values/strings.xml` when a string is absent in a locale-specific file.
- The app reads the user's `preferred_locale` from the API session token and applies it via `AppCompatDelegate.setApplicationLocales()`.

### iOS

- Localisation files reside in `en.lproj/Localizable.strings`, `fr.lproj/Localizable.strings`, and `sw.lproj/Localizable.strings` under the main bundle.
- All UI strings are referenced using `NSLocalizedString("key", comment: "")` or the SwiftUI `Text("key")` initialiser with the `tableName` parameter.
- iOS falls back to the `en.lproj` bundle automatically when a string is absent in the selected locale.
- The app overrides the system locale using `Bundle` subclassing to honour the user's in-app `preferred_locale` setting independently of the device locale.

## Translation Principles

1. Translation is contextual, not word-for-word. A string that reads naturally in English may require reframing in French or Kiswahili to convey the same clinical meaning. Literal translations are rejected.
2. Clinical severity labels (`Fatal`, `Serious`, `Warning`, `Info`) are rendered in the clinician's active UI language at all times. These labels are never auto-translated mid-workflow.
3. Kiswahili summaries targeting patient-facing content (AI Patient Plain-Language Summary, patient portal notes) are written at a lower reading level than French or English equivalents, reflecting the health literacy distribution of the primary Kiswahili-speaking patient population.
4. A native-speaker clinician fluent in the target language must review all clinical string translations before they are merged into any release branch. Non-clinical strings (navigation labels, system messages) require only a fluent native speaker.
5. No string may enter production with an `[I18N-GAP]` flag unresolved. Unresolved gaps are release blockers.

## Per-Module Localisation Flags

Flags are listed only where cultural or clinical adaptation is non-obvious.

| Module | Flag | Guidance |
|--------|------|----------|
| Maternity | `maternity.registration.birth_attendant_label` | `fr`: *accoucheuse*, `sw`: *mkunga* — different professional categories per country context; do not use generic "attendant" |
| Billing | Number formatting | `fr`: 1 500 000 UGX (space thousands separator per French convention); `en` and `sw`: 1,500,000 UGX (comma separator) |
| AI Patient Plain-Language Summary | Reading level | Kiswahili summaries target a lower Flesch–Kincaid reading level than French; both target lower than English clinical equivalents |
| OPD Triage | `opd.triage.chief_complaint_label` | `sw` value: *malalamiko makuu* (contextually accurate); literal translation *malalamiko ya kwanza* is clinically ambiguous and is prohibited |

## Gap Management Protocol

When a string key exists in `en` but is absent from `fr` or `sw`:

1. The build log emits `[I18N-GAP: <key>]`.
2. The CI pipeline flags the build as a warning (not a failure) in development branches.
3. The CI pipeline fails the build in the `release` branch — unresolved `[I18N-GAP]` tags are a release blocker.
4. The gap is assigned to the translation queue in the project issue tracker with the label `i18n-gap`.
5. The English fallback string is rendered in production until the translation is approved and merged.

No `[I18N-GAP]` tag is removed from the build log without a human-approved translation merged and verified by a native-speaker reviewer.
