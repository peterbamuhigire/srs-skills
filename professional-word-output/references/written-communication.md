## Core Communication Principles

> "Always assume the reader knows nothing about the subject. The writer's responsibility is to explain concepts clearly." — *Clear Written Communication*

These are non-negotiable regardless of document type:

1. **Define the objective first.** Before writing a single word, state in one sentence what this document must achieve. Every section, paragraph, and sentence either serves that objective or it does not belong.
2. **Write for the reader, not the writer.** The writer already understands the content. The question is always: will someone who doesn't know this understand it?
3. **Adapt vocabulary to audience knowledge level.** Jargon is appropriate with expert audiences. Use plain language for mixed or non-technical audiences.
4. **Never convey more information than necessary.** Completeness is not the same as length. A 2-page document that achieves the objective outperforms a 20-page document that buries it.

## Document Structure

Every professional document follows a three-part structure:

### 1. Introduction
Sets context and purpose. Answers: *Why does this document exist? What will the reader gain?*

- State the document's purpose in the first paragraph
- Identify the audience and their expected knowledge level
- Briefly outline what follows (signpost)
- Keep it short — the introduction is not the document; it is the entry point

### 2. Development (Body)
Provides the main content, evidence, and reasoning. Each section:
- Pursues exactly one communication objective
- Opens with a topic sentence stating what the section covers
- Supports that topic with evidence, examples, or data
- Closes with a transition or summary sentence

### 3. Conclusion
Summarises key ideas and (if applicable) states required action. Answers: *What should the reader remember? What should they do next?*

- Never introduce new information in the conclusion
- Restate the single most important point
- Give a clear call to action if the document requires one

## Writing for Clarity

### Sentence Construction

- **Short sentences.** 15–20 words average. Long sentences undermine reader attention and dilute meaning.
- **One idea per sentence.** When a sentence contains two ideas joined by "and" or "but", consider splitting it.
- **Active voice.** "The board approved the proposal" is stronger than "The proposal was approved by the board."
- **Present tense where possible.** "The system processes requests" vs "The system will process requests" — present tense creates confidence.
- **Affirmative sentences.** "Submit the form by Friday" is clearer than "Do not delay submission past Friday."

### Paragraph Construction

- **One topic per paragraph.** When you can't summarise the paragraph in one sentence, split it.
- **Short paragraphs.** 3–5 sentences maximum for business writing. Longer paragraphs belong in academic papers.
- **Logical connectors at paragraph openings:** "However," / "Therefore," / "As a result," / "In addition," / "For example,"
- **Avoid orphaned paragraphs** — a paragraph with only one sentence usually belongs merged or expanded.

### Conciseness Rules

| Instead of | Write |
|---|---|
| "In order to" | "To" |
| "Due to the fact that" | "Because" |
| "At this point in time" | "Now" |
| "In the event that" | "If" |
| "It should be noted that" | (Delete — just state the fact) |
| "A significant number of" | "Many" or give a number |
| "Utilise" | "Use" |
| "Facilitate" | "Help" or "Enable" |

## Tone and Register

### Business / Corporate Documents

- **Neutral, formal, and precise.** Not cold or bureaucratic — calm and authoritative.
- Avoid contractions ("it's", "don't") in formal reports and proposals.
- Use contractions in emails, internal memos, and conversational documentation.
- **Never use vague adjectives** (fast, robust, seamless, intuitive, powerful). Replace with measurable specifics:
  - Bad: "The system is fast."
  - Good: "The system processes 500 transactions per second."

### Technical / Specification Documents

- Define every acronym on first use: "Application Programming Interface (API)"
- Use consistent terminology — never alternate between "user," "client," and "customer" for the same concept
- Use numbered lists for sequential steps; bullet lists for unordered items
- Use imperative voice for instructions: "Click Save." not "You should click Save."

### Proposals / Persuasive Documents

- Lead with the benefit to the reader, not features of the product/service
- Use social proof, evidence, and data to support claims
- Structure around the reader's problem → your solution → evidence → call to action
- Avoid self-congratulatory language ("We are the leading…", "Our world-class…")

## Sentence and Paragraph Craft

### The Inverted Pyramid

Start with the most important information. Readers scan before they read. If the conclusion or key finding is buried on page 4, most readers never reach it.

```
Most important → Supporting detail → Background
```

Apply this at every level: document, section, paragraph, sentence.

### Use Examples Generously

Abstract claims require concrete illustration. After any principle or rule, provide an example. After any instruction, show the expected result.

### Signposting

Readers need to know where they are in a long document. Use:

- **Section headings** — obvious but essential
- **Summary sentences** at section ends: "The next section covers..."
- **Cross-references**: "See Section 4.2 for configuration requirements."
- **Numbered lists** when sequence matters

## Anti-Patterns to Eliminate

These are the most common writing failures in AI-assisted professional documents:

| Anti-pattern | Why it fails | Fix |
|---|---|---|
| "Delve into" | Pretentious, AI-signature phrase | "Explore," "examine," or just state the thing |
| "Leverage" | Overused buzzword | "Use" |
| "Robust" | Meaningless without metrics | Specify what makes it robust |
| "Seamlessly" | Unverifiable marketing claim | Delete or specify |
| "In today's digital landscape" | Empty opener | Start with the actual point |
| "It is worth noting that" | Filler | Delete — just state the note |
| "Moving forward" | Vague | Specify what and when |
| Passive voice everywhere | Hides responsibility | Use active voice |
| Acronyms undefined | Excludes readers | Define on first use |
| All headings as questions | Can work, but overused in AI output | Mix statement headings with questions |
| Numbered lists for non-sequential items | Implies order where none exists | Use bullet lists |
| Bullet lists for sequential steps | Loses the sequence signal | Use numbered lists |
| Triple nesting (bullets inside bullets inside bullets) | Creates visual noise | Flatten to 2 levels max |
